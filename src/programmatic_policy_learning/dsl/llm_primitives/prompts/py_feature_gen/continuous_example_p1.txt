You are generating boolean feature functions for a continuous KinDER environment.

Environment:
- Task is a continuous control environment from KinDER.
- A robot acts in a continuous 2D setting.
- Observation `s` is a numeric state vector.
- Action `a` is a numeric control vector.
- `p1` means one narrow passage/obstacle.

- Demonstrations are expert trajectories over `(s, a, s')`.

## INPUT: DEMONSTRATIONS
${DEMONSTRATION_BACKGROUND}

You are given the following expert demonstrations:

${DEMONSTRATION_DATA}

Task:
Propose a diverse set of boolean features `f(s, a)` for distinguishing expert actions from alternative candidate actions.

Goal:
Produce features that are useful for learning a policy from demonstrations, not just features that describe the state or action in a shallow way.

A good feature set should:
- generalize across episodes,
- capture reusable structure in the relationship between state and action,
- help separate good and bad candidate actions within the same state,
- support compositional decision rules,
- avoid depending on exact coordinates or episode-specific values.

The feature set should be functionally diverse.
Include a balanced mix of:
- state-based relational features,
- action-based structural features,
- state-action interaction features,
- progress-related features,
- safety or constraint-respecting features,
- conditional features that are only meaningful under certain state conditions,
- comparative features that express tradeoffs between multiple tendencies.

Prefer features that help answer questions like:
- what kind of situation the agent is currently in,
- whether the action is appropriate for that kind of situation,
- whether the action makes meaningful progress,
- whether the action respects important constraints,
- whether the action reflects a useful tradeoff rather than optimizing only one simple quantity.

Prefer features that can participate in compositional policies:
- some features should characterize the current regime or situation,
- some features should characterize whether the action is suitable for that regime,
- some features should become useful mainly when combined with other features in a logical rule.

Include features at multiple abstraction levels:
- simple numeric relations,
- medium-level relational or geometric patterns,
- more selective conditional-appropriateness tests.

Avoid weak feature sets dominated by:
- single-dimension sign checks,
- raw threshold checks on one coordinate,
- unconditional “action reduces one scalar quantity” tests,
- many small variations of the same inequality,
- features that are almost always true or almost always false,
- features that merely restate each other,
- brittle features tied to exact demonstration values.

Before returning the final set, internally filter out features that are:
- redundant,
- near-duplicates,
- weakly discriminative,
- too brittle to generalize,
- too trivial to help a downstream logical learner.

Return at least 50 features.

Output:
Return only valid JSON:
{
  "features": [
    {"id": "f1", "name": "f1", "source": "def f1(s, a):\\n    ..."}
  ]
}

Rules:
- No placeholders.
- No markdown.
- No commentary.
- No imports.
- Each feature must return `True` or `False`.
- Treat `s` and `a` as raw arrays only.
- Access them only with numeric indices from the field guide.
- Each feature must be a standalone function.
- Use only information available in `s` and `a`.
- Prefer relational and reusable features over brittle or highly specific ones.
- Use as few arbitrary numeric constants as possible.
- Avoid near-duplicate features.
- Avoid trivial features that are almost always true or almost always false.
