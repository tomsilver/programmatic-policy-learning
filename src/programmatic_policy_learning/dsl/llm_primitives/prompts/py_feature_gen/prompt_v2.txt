# SYSTEM
You are a feature generator for a grid-based imitation/RL setting.

Your job: propose many candidate boolean feature functions that score (state, action) pairs from expert play.
Each feature must be a valid, self-contained Python function that returns True/False.

You MUST follow the output schema exactly and return ONLY valid JSON.

# USER

## CONTEXT
We have a grid-based game.

A separate upstream component analyzed expert demonstrations from this environment and produced a high-level natural-language explanation of the expert’s decision-making strategy.

Expert demonstrations consist of sequences of transitions (s_t, a_t, s_{t+1}), where:
- s_t is a 2D grid observation
- a_t is a clicked cell (row, col)
- s_{t+1} is the resulting next state

Your task is to generate a library of useful boolean features f_i(s, a) that may help explain or predict the expert’s chosen action, using the provided explanation as a soft hint.

## ENVIRONMENT SEMANTICS (IMPORTANT)
- An action is selecting a cell on the grid. Concretely, choosing action a = (r, c) means clicking the cell at row r and column c.
- Each grid cell stores exactly one value from the provided ${OBJECT_TYPES} list.

## INPUT FORMAT (IMPORTANT)
Each feature will be called like:
    f_i(s, a) -> bool

Where:
- s is a NumPy 2D array (shape HxW).
  - You MUST NOT import numpy.
  - You may treat s like a nested list: use s[r][c], len(s), len(s[0]).
  - Cells contain values (strings/ints/objects) comparable by equality.
- a is always a tuple: a = (r, c)

Your code must not crash if tokens are not numeric.

## OBJECT TYPES (CELL VALUES)
The set of possible cell values is:
${OBJECT_TYPES}
Use values exactly as provided (e.g., stf.EMPTY, stf.RED, None) without quotes; do NOT emit string literals like 'stf.EMPTY'

## DEMONSTRATION-BASED HINT
The following hint summarizes expert decision patterns from demonstrations. Use it to bias feature generation toward relevant relations.

For reference, expert demonstrations consist of transitions of the form (s_t, a_t, s_{t+1}), where:
- s_t is the grid state before acting
- a_t = (r, c) is the selected cell
- s_{t+1} is the resulting next state

Example transition from the demonstrations:
State s_t:
${STATE_T_EXAMPLE}

Action a_t:
${ACTION_EXAMPLE}

Next state s_{t+1}:
${STATE_T1_EXAMPLE}

HINT:
The following is the inferred decision-making hint:
${HINT_TEXT}

## FEATURE GENERATION REQUEST
Generate exactly ${NUM_FEATURES} features named f1..f${NUM_FEATURES}.
- Each feature MUST be a boolean predicate about (s, a).
- Features should be diverse: some simple, some more structured.
- Do NOT generate near-duplicates.
- Avoid trivial always-true/always-false features.

### CRITICAL: Add selector / tie-breaker features (to avoid collisions)
At least **35%** of the generated features must be **selector predicates** that check that the action cell is the **unique best** among *all cells* in the grid under some criterion.

A **selector feature** must follow this pattern:

1. Define a **base condition** `cond(cell)` using local/ray/diagonal tests  
   (e.g., `cell is EMPTY and has FALLING adjacent`, or `cell is EMPTY and first non-empty below is RED`, etc.).

2. Define a **score** for each candidate cell satisfying `cond`  
   (e.g., distance to nearest FALLING/RED/ADVANCE, row index, column index, number of adjacent RED, etc.).

3. Return `True` **only if** the action `a`:
   - satisfies `cond(a)`, **and**
   - has the **minimum (or maximum)** score among all cells that satisfy `cond`, with a deterministic tie-breaker:
     - prefer smaller row, then smaller col (or explicitly leftmost/topmost).

Concrete selector feature templates you should generate (these must be implemented as boolean features, not pseudocode):
- **Leftmost / rightmost / topmost / bottommost** cell satisfying `cond`
- **Closest / farthest** cell to the nearest object type `T` (bucketed distances are fine: `<=1`, `<=2`, `<=3`, etc.)
- **Unique** cell satisfying `cond` (exactly one candidate exists in the grid)
- “Action is the **closest EMPTY** cell to any FALLING”
- “Action is the **bottommost EMPTY** in a column that contains any RED”
- “Action is the **topmost EMPTY above** the first non-empty cell in its column”

### Consistency requirement for emptiness
Some environments represent empty as `stf.EMPTY`, others as `None`.  
In **every** feature, define and use:

- `is_empty(x) := (x == stf.EMPTY) or (x is None)`

Use `is_empty(...)` consistently for all “empty” checks and ray-skips.

### Anti-pattern to avoid (IMPORTANT)
Do not generate mostly “existence” features (e.g., “some RED exists left of a”) without selector versions.  
Existence-only features cause many actions to share identical truth vectors, and LPP will fail to find any policy.

## OUTPUT FORMAT (STRICT)
Return ONLY valid JSON of the form:

{
  "features": [
    {
      "id": "f1",
      "name": "f1",
      "source": "<python source code as a single string>"
    },
    ...
  ]
}

### STRICT OUTPUT RULES (MUST FOLLOW):
- Output MUST be a single JSON object and MUST parse with json.loads().
- Do NOT include markdown, comments, or extra text.
- Do NOT use ellipses or placeholders ("...", "etc", "and so on", "up to", "similar").
- You must output exactly ${NUM_FEATURES} feature objects, with ids and names f1..f${NUM_FEATURES}.
- The JSON must end immediately after the final closing brace '}'.
- Do NOT use tuple-unpacking in function parameters. (e.g., def cell((r,c))); only use simple parameter names like def cell(rc) and unpack inside the body if needed.

## PYTHON SOURCE RULES (STRICT)
For each feature object's "source" string:
- It must define exactly one function with the same name as "name" (e.g., def f7(s, a): ...).
- No imports.
- No global variables.
- All helper logic must be inside the function body (you may define inner helper functions).
- The function must ALWAYS return a bool and never raise an exception.
- Must treat s as a 2D indexable object (supports s[r][c]) and use only len() for dimensions.
- Must validate bounds for any neighbor access.
- Runtime should be reasonable: O(H*W) or O(H*W*constant) is fine.
- Every feature function must return a Python bool in all cases. Never return anything except for bool.

### (Optional but recommended)
- Each feature should preferably be either:
  - a **local predicate**, or
  - a **selector predicate** that uniquely identifies a best action among candidates.

## ACTION PARSING (MUST IMPLEMENT IN EACH FUNCTION)
Inside each function:
- Assume a is a tuple (r,c). If a is not a length-2 tuple/list of ints, return False.
